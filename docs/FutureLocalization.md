# Localization Guide

This document is the canonical reference for all localization work in GT5-Unofficial.
If you are touching the lang file, adding translatable strings in Java, or doing a localization
rework, read this first.

---

## 1. Java API - Which Method to Use

### 1.1 `StatCollector.translateToLocal(key)`
Use for any text that is **rendered on the client side**: item tooltips, item names, GUI labels,
WAILA/NEI output, anything inside a `@SideOnly(Side.CLIENT)` context.

```java
import net.minecraft.util.StatCollector;

String name = StatCollector.translateToLocal("gt.item.item_name.name");
```

### 1.2 `StatCollector.translateToLocalFormatted(key, args...)`
Same as above but with format substitution. Use when the translated string contains `%s` or `%d`
placeholders.

```java
String desc = StatCollector.translateToLocalFormatted("gt.item.item_name.tooltip", 16000);
// lang: gt.item.item_name.tooltip=Capacity: %d L
```

**Never use these two methods in code that runs server-side for chat output.** On a dedicated
server the lang file may not be loaded, so the key will be returned as-is instead of translated.

### 1.3 `GTUtility.sendChatTrans(player, key, args...)` - Chat Messages
Use for sending a single translated message to a player. Internally wraps `ChatComponentTranslation`,
which ships the raw key over the network and lets the **client** translate it. This works correctly
in both singleplayer and multiplayer.

```java
import gregtech.api.util.GTUtility;

// Server-side (inside a TileEntity, Command, Event handler, etc.)
GTUtility.sendChatTrans(player, "gt.chat.error.machine_broken", machineName);
// lang: gt.chat.error.machine_broken=Your machine %s has broken!
```

> **Singleplayer vs. Multiplayer note:**
> `sendChatTrans` is safe in both environments. In singleplayer the integrated server sends the
> `ChatComponentTranslation` to the local client. On a dedicated server it is sent over the network.
> Either way, translation happens on the client in the player's own language.
> Never pre-translate the string with `StatCollector` before passing it to chat - this breaks
> multiplayer because translation would happen with the server's (possibly missing) lang file.

### 1.4 `GTUtility.sendChatComp(player, component)` - Complex Chat Messages
Use when a single message needs multiple independently-formatted parts (e.g. a colored item name
followed by plain text). Build an `IChatComponent` manually and pass it in.

```java
import net.minecraft.util.ChatComponentText;
import net.minecraft.util.ChatComponentTranslation;
import net.minecraft.util.ChatStyle;
import net.minecraft.util.EnumChatFormatting;

IChatComponent msg = new ChatComponentTranslation("gt.chat.info.found_item")
    .appendSibling(
        new ChatComponentText(itemName)
            .setChatStyle(new ChatStyle().setColor(EnumChatFormatting.GREEN)));
GTUtility.sendChatComp(player, msg);
```

### 1.5 MUI2 - `IKey.lang(key, args...)`
For widgets inside **ModularUI 2** GUIs (`modularui2` package):

```java
import com.cleanroommc.modularui.api.IKey;

// Simple key
IKey.lang("gt.gui.my_label").asWidget()

// Key with placeholders
IKey.lang("gt.gui.tick_rate", tickRate)

// As tooltip line
tooltip.addLine(IKey.lang("gt.gui.my_tooltip"))
```

## 2. Text Formatting

Use `EnumChatFormatting` constants. Concatenate them directly with strings.

```java
import net.minecraft.util.EnumChatFormatting;

String text = EnumChatFormatting.RED + "Warning: " + EnumChatFormatting.WHITE + "do not touch";
```

In lang file values you may use `§` codes directly (e.g. `§c` for red, `§r` for reset), but
prefer generating formatting via `EnumChatFormatting` in Java to avoid hard-coded codes.

---

## 3. Format Placeholders

| Placeholder | Use for |
|---|---|
| `%s` | String values (item names, player names, any text) |
| `%d` | Integer / whole number values |
| `%f` or `%.2f` | Floating point values (use sparingly, prefer formatting before passing) |
| `%1$s`, `%2$s` | Positional placeholders when the order may differ between languages |

Examples in the lang file:

```properties
gt.capacity.tooltip=Capacity: %d L
gt.chat.welcome=Welcome, %s!
gt.transfer.tooltip=Transfer rate: %1$s / %2$s ticks
```

## 4. Lang File Key Naming

### 4.1 Prefix
All new keys must use the `gt.` prefix.

```
gt.<category>.<subcategory>.<identifier>
```

**Exception — Creative Tab:** Item group (creative tab) keys are generated by Minecraft/Forge
using the format `itemGroup.<registryName>.<tabName>`. These do not use the `gt.` prefix.

```properties
itemGroup.GregTech.Main=Main
itemGroup.GregTech.Materials=Materials
```

### 4.2 Case & Separators
- All lowercase except the `gt` prefix.
- Use `.` as separator between segments.
- Use `_` as separator within a segment word (snake_case within a segment).

```properties
# Good
gt.chat.error.machine_broken=Your machine %s has broken!
gt.gui.cover.fluid_filter.title=Fluid Filter

# Bad - wrong case, wrong separator
gt.Chat.MachineBroken=...
gt.chat.machine-broken=...
```

### 4.3 Suffix Conventions
Suffixes always come **last** in the key. `.tooltip` in particular must never appear in the
middle of a key - it is always the final segment.

| Suffix | Meaning |
|---|---|
| `.name` | Display name of an item or block |
| `.tooltip` | Tooltip line(s). Always the last segment. |
| `.tooltip.N` | Numbered tooltip lines when there are multiple (N = 1, 2, 3…). Still last. |
| `.title` | GUI window / panel title |
| `.button.*` | Button label |
| `.label.*` | Static text label in a GUI |
| `.info.*` | Informational line in GUI or WAILA |
| `.structure_error` | Multiblock structure error / reason it isn't formed |
| `.shutdown_reason` | Reason a machine stopped or cannot run |
| `.recipe_result` | Result or status text of a recipe or processing step |

```properties
# Good - .tooltip is always at the end
gt.item.my_item.tooltip=Some tooltip text.
gt.item.my_item.tooltip.1=First extra line.
gt.item.my_item.tooltip.2=Second extra line.
gt.gui.cover.fluid_filter.title=Fluid Filter

# Bad - .tooltip buried in the middle
gt.item.my_item.tooltip.name=...
gt.item.my_item.name.tooltip=...
```

### 4.4 `\n` vs Numbered `.tooltip.N` Keys
Both approaches produce multi-line tooltips. Choose based on whether the lines are always shown
together or can vary independently.

| Approach | When to use |
|---|---|
| `\n` inside one key | Lines are always shown as a fixed block and will never be shown separately |
| `.tooltip.N` separate keys | Lines may be shown conditionally, reordered, or translated differently per language |

```properties
# \n — fixed multi-line block, lines always appear together
gt.item.cable.tooltip=Max Voltage: %s\nMax Amperage: %s\nLoss/m/A: %s

# .tooltip.N — independent lines, easier to extend or conditionally include
gt.item.battery.lv.tooltip=Stores up to %d EU.
gt.item.battery.lv.tooltip.1=Right-click to check charge.
gt.item.battery.lv.tooltip.2=Discharge rate: %d EU/t.
```

### 4.5 File-Scoped Keys
When a lang key is **only used inside a single source file**, include the class name as a scope
segment immediately after the category. Strip the common prefixes (`MTE`, `GT`, `GT5U`) and
write the remainder in `camelCase`.

File-scoped keys still go under their **normal functional section** in the lang file (GUI keys
under `#### GUI`, info data under `#### Scanner & Info Data`, etc.). The class scope is only
part of the key name — it is not a separate section.

```
gt.<category>.<classScope>.<identifier>
```

| Class | Scope segment |
|---|---|
| `MTEBasicBatteryBuffer` | `basicBatteryBuffer` |
| `MTEMultiBlockBase` | `multiBlockBase` |
| `GTCoverFluidFilter` | `coverFluidFilter` |
| `CoverConveyor` | `coverConveyor` |

**Example** - keys in `MTEBasicBatteryBuffer.java`:

```properties
gt.infodata.basicBatteryBuffer.stored_items=Stored: %s / %s EU
gt.infodata.basicBatteryBuffer.average_input=Avg. Input: %s EU/t
```

If the same key is needed by **more than one file**, drop the class scope segment and use only
the category, making the key shared:

```properties
# shared across multiple machine files
gt.infodata.average_input=Avg. Input: %s EU/t
```

## 5. Lang File Structure & Section Headers

Comments in `.lang` files start with `#`. Use the following heading depth to organize sections.
This structure mirrors Markdown heading levels and makes the file easier to navigate.

```
#### (4 hashes)  - Top-level / main group
### (3 hashes)   - Sub-group level 1
## (2 hashes)    - Sub-group level 2
# (1 hash)       - Sub-group level 3 (most commonly used for fine-grained sections)
```

### 5.3 Where to Insert New Keys

**All new keys go at the bottom of the file**, below the designated block:

```properties
# ================================================================================
# ================================================================================
# New lang system according to docs\FutureLocalization.md
# Insert new keys under
# ================================================================================
# ================================================================================
```

Place your key under the correct section header. If the section does not exist
yet, create it using the appropriate heading depth.

## 6. Summary Table - Which Method When

| Situation | Method |
|---|---|
| Item / block display name (client) | `StatCollector.translateToLocal(key)` |
| Tooltip lines (client) | `StatCollector.translateToLocal(key)` / `translateToLocalFormatted` |
| NEI / WAILA display text | `StatCollector.translateToLocal(key)` |
| Chat message to player (server code) | `GTUtility.sendChatTrans(player, key, args...)` |
| Complex chat message (multi-component) | `GTUtility.sendChatComp(player, component)` |
| MUI2 GUI widget text / tooltip | `IKey.lang(key, args...)` |
| MUI2 standalone text widget | `IKey.lang(key).asWidget()` |

---

## 7. Checklist for Adding a New Translatable String

1. [ ] Determine the correct method from the table above.
2. [ ] Choose the key using `gt.<category>.<subcategory>.<identifier>` convention.
3. [ ] Add the key at the bottom of `en_US.lang` under the correct section header.
4. [ ] Use `%s` / `%d` placeholders where values are substituted; never concatenate raw strings.
5. [ ] Use `EnumChatFormatting` constants for color in Java. Hard-coded `§` codes are acceptable in the lang file itself, and also in Java when building multi-color sequences (e.g. rainbow strings) where constants would be impractical.
6. [ ] For chat messages, always use `sendChatTrans`; never pre-translate on the server.
